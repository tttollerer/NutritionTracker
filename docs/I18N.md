# i18n — Single-Locale-Entscheidung (Stand Abschluss, Paket 15)

## Entscheidung

**Deutsch (`de`) ist die einzige ausgelieferte Locale.** Das ist plankonform:
PLAN.md §11 (Phase 0) definiert „DE als Startsprache". Ein `en.json`-Vollausbau
wurde im Abschlussplan bewusst verschoben (docs/ABSCHLUSSPLAN.md, Nicht-Ziele) —
er wäre reiner Übersetzungsaufwand ohne strukturelle Änderung.

## Warum die Struktur trotzdem mehrsprachig-ready ist

- **Alle UI-Texte liegen in `src/i18n/locales/de.json`**; Komponenten rendern
  ausschließlich über `t('…')`. Auch Registries außerhalb von React tragen nur
  i18n-Keys (z. B. `THEMES[].labelKey` in `src/lib/themes.ts`), die Auflösung
  passiert in der UI-Schicht.
- **Keine String-Konkatenation für Sätze:** Variablen laufen über
  i18next-Interpolation (`{{name}}`, `{{count}}` …), Plurale über die
  i18next-Suffixe (`_one`/`_other`, z. B. `add.copiedYesterday`).
- **lib-Code wirft Fehlercodes, keine übersetzten Texte** (`src/lib/apiError.ts`
  mappt Codes → `errors.*`; `src/lib/image.ts` wirft `ImageError` mit Code).
- Eine weitere Sprache bedeutet: `locales/en.json` anlegen, in
  `src/i18n/index.ts` unter `resources` registrieren, `lng` umschaltbar machen.

## Bewusste Ausnahme: Katalog-Lebensmittelnamen

`src/lib/foodCatalog.ts` löst die Namen aus `catalog.foods.*` einmalig pro
Zugriff über `i18n.t()` auf und liefert sie als fertige Strings, weil
`CatalogFood.name` beim Quick-Log als `FoodItem.name` in Dexie **persistiert**
wird. Persistierte i18n-Keys würden historische Einträge und Backups an
Key-Namen der jeweiligen App-Version koppeln. Bei einem späteren Sprachwechsel
werden neue Logs in der aktiven Sprache angelegt; bestehende Einträge behalten
den Namen, mit dem sie geloggt wurden.
